# 电子钟_纯软件调试要点与详细方案

## 1. 纯软件调试总目标

在不依赖真实 RTC、按键、触摸屏、LED、蜂鸣器、红外遥控器和后备电池的前提下，尽量验证电子钟 application 核心逻辑、输入事件处理逻辑和输出状态逻辑。

纯软件调试的核心入口：

- 仿真开关：`03_软件工程/电子钟训练工程/bsp/sim_debug_config.h` — 宏 `APP_CLOCK_SIM_ENABLED` (line 9) 和 `APP_CLOCK_SIM_AUTO_RUN` (line 17)
- 主入口切换：`03_软件工程/电子钟训练工程/User/main.c` — `#if APP_CLOCK_SIM_ENABLED` (line 14)
- 仿真入口：`03_软件工程/电子钟训练工程/application/app_clock_sim.c` — `AppClockSim_Main()` (line 238)
- 状态快照接口：`03_软件工程/电子钟训练工程/application/app_clock.c` — `App_Clock_GetDebugSnapshot()` (line 133)

所有文件路径均相对于工作区根目录 `D:\Electronics Design Contest\04_个人软件实验区\`。

## 2. 所有需要调试的要点

### 2.1 入口与模式切换

- `APP_CLOCK_SIM_ENABLED` 是否能正确切换正式路径和仿真路径。
- `APP_CLOCK_SIM_AUTO_RUN` 是否能正确控制"自动回归"还是"停在入口等待人工调试"。
- 仿真模式下是否绕过了真实 `LCD/Touch/IR/Storage/UART` 初始化。

### 2.2 时基与 RTC

- `Drv_Systick_IrqHandler()` 是否每次推进 1ms。
- `App_Clock_Task1ms()` 是否同时推进 `Drv_Key_Task1ms()` 和 `Drv_Rtc_Task1ms()`。
- 软件 RTC 是否能从指定初始时间开始计时。
- `Drv_Rtc_SetTime()` 写入后，`Drv_Rtc_GetTime()` 是否立即反映。
- 1 秒推进后，时分秒是否正确进位。
- `23:59:59 -> 00:00:00` 是否正确回卷。

### 2.3 物理按键驱动

- `click` 是否在按下稳定、释放稳定后产生。
- `long press` 是否在 800ms 左右触发。
- `repeat` 是否在长按后按设定周期持续触发。
- 去抖逻辑是否防止误触发。
- 仿真注入接口 `Drv_Key_SimSetLevel()` 是否只影响仿真模式。

### 2.4 application 核心状态机

- `RUN -> SET_TIME_HOUR`
- `SET_TIME_HOUR -> SET_TIME_MINUTE -> SET_TIME_SECOND -> RUN`
- `RUN -> SET_ALARM_HOUR -> SET_ALARM_MINUTE -> RUN`
- 编辑值加一、减一是否满足范围约束。
- 小时是否限制在 `0~23`，分钟/秒是否限制在 `0~59`。
- 编辑过程中是否持续刷新 `last_activity_tick`。
- 长按取消编辑后，RTC 当前时间是否保持不变。
- 保存编辑后，RTC 或闹钟参数是否更新。

### 2.5 编辑超时

- 无操作 10 秒后是否从任意编辑态退回 `RUN`。
- 超时退出后是否不写入 RTC。
- 超时退出后 UI 与指示状态是否恢复到 `RUN`。

### 2.6 闹钟逻辑

- 闹钟使能开/关是否正确切换。
- 闹钟在 `hour/minute` 匹配且 `second == 0` 时是否只触发一次。
- 响铃期间 LED / 蜂鸣器状态是否正确。
- 响铃期间按键是否能停止闹钟。
- 响铃超时是否能自动停止。
- **注意：当前实现的响铃超时值是 30 秒（宏 `APP_ALARM_RING_MS = 30000UL`），不是讲义要求的 60 秒，必须单独标注。**

### 2.7 触摸逻辑键

- 当前代码中，触摸逻辑键和物理键最终共用同一套核心状态机。
- `TIME / ON-OFF / STATE / ALARM` 在 `RUN` 态的语义是否正确。
- `NEXT / BACK / PLUS / MINUS` 在编辑态的语义是否正确。
- 触摸长按 `PLUS / MINUS` 是否能连发。
- 触摸路径是否与物理键路径行为一致。

### 2.8 红外遥控映射

- 红外事件是否能映射到与触摸相同的逻辑动作。
- `VOL+/VOL-` 是否映射为连加/连减动作。
- `CH-` 是否切换闹钟使能。
- `100+` 是否切换静音。
- 当前尚未实现"数字键直接设置时间/闹钟"，需要单独标注为未覆盖项。

### 2.9 输出观测

- `BSP_LED_GetMask()` 是否能反映当前 LED 输出状态。
- `BSP_Buzzer_GetState()` 是否能反映当前蜂鸣器输出状态。
- 编辑态、运行态、闹铃态的 LED 颜色是否符合设计。
- 静音打开后闹铃期间蜂鸣器是否关闭但 LED 仍闪烁。

### 2.10 参数保存

- `AppClockStorage_Save()` 是否能写入闹钟时间、闹钟使能、静音使能。
- `AppClockStorage_Load()` 是否能恢复这些参数。
- 当前仿真入口默认绕过 storage 写入，因此掉电保持尚未进入自动脚本回归。

### 2.11 UI 与显示

- `AppClockUi_Render()` 是否正确显示当前主时间、闹钟时间、模式文本、字段文本。
- `RUN / TIME SET / ALARM SET / RINGING` 的 UI 标签是否正确。
- 纯软件调试只能验证"渲染调用逻辑"和显示文本来源，不能真正证明 LCD 视觉效果。

### 2.12 仍需真实硬件验证的点

- 硬件 RTC + 后备电池断电走时
- 真实触摸精度与坐标映射
- 真实红外接收头时序鲁棒性
- 真实 LCD 横屏显示效果
- 蜂鸣器音量、按键音、整点提示
- 实际接线、电平极性、板级资源冲突

## 3. 最重要的调试部分

application 核心状态机中的改时流程。

原因：

- 这是所有输入路径的汇合点。物理键、触摸逻辑键、红外最终都要落到这套状态机。
- 这是讲义里最核心、最容易出错、最影响后续闹钟/保存/界面逻辑的部分。
- 一旦这里有错，按键驱动、触摸驱动、RTC 驱动都可能看起来"像是坏了"，但根因其实在状态机。

## 4. 详细调试方案：改时状态机

### 4.1 调试目标

验证以下路径：

1. 进入改时
2. 编辑字段切换
3. 加 1
4. 减 1
5. 长按取消
6. 10 秒无操作超时退出

说明：

- 当前现成脚本已经覆盖"取消路径"和"超时路径"。
- "保存路径"（SET_TIME_SECOND 后自动保存）暂未在自动脚本中单列，但核心状态机的保存代码已经存在。

### 4.2 参数设置

在 `03_软件工程/电子钟训练工程/bsp/sim_debug_config.h` 中确认：

```c
#define APP_CLOCK_SIM_ENABLED    1U
#define APP_CLOCK_SIM_AUTO_RUN   1U
```

含义：

- `APP_CLOCK_SIM_ENABLED = 1U`：进入纯软件仿真模式，绕过真实硬件初始化。
- `APP_CLOCK_SIM_AUTO_RUN = 1U`：上电后自动执行回归脚本 `AppClockSim_RunAll()`。如果改为 `0U`，则停在 `AppClockSim_Main()` 的死循环中等待人工单步。

### 4.3 Keil 操作步骤

#### 步骤 0：前置配置（首次调试必须执行，后续可跳过）

Keil CPU Simulator 不模拟 STM32 外设。若不配置，程序会在启动阶段因访问 RCC 寄存器而报 `error 65: access violation`。

下面两项已在本工程中完成：

**(a) system_stm32f10x.c 修改**

`Start/system_stm32f10x.c` 已修改——在 `SystemInit()` 函数开头加入仿真模式跳过逻辑：

```c
#if APP_CLOCK_SIM_ENABLED
  SCB->VTOR = FLASH_BASE | VECT_TAB_OFFSET;
  SystemCoreClock = HSI_VALUE;
  return;
#endif
```

这确保在仿真模式下，启动代码不会尝试配置 HSE/PLL 时钟，避免在等待硬件 ready 标志位的 `while` 循环中死锁。

**(b) 配置 Keil Debug Initialization File**

1. 在 Keil 菜单中：`Project -> Options for Target '...' -> Debug`
2. 确保右上角选择 `Use Simulator`
3. 在 `Initialization File:` 输入框中，点击右侧 `...` 按钮，选择：
   `03_软件工程/电子钟训练工程/debug_sim.ini`
4. 点击 OK 保存

`debug_sim.ini` 会执行 `MAP` 命令，将 STM32F103 的外设地址空间（0x40000000-0x40030000）映射为可读写内存。这作为安全网——即使某些代码路径意外触碰了外设寄存器，也不会触发 access violation。

#### 步骤 1-8：正式调试

1. 打开工程 `Project.uvprojx`。
2. 确认 `sim_debug_config.h` 中上述两个宏的值。
3. `Rebuild` 工程，确保无编译错误。
4. 打开源文件：
   - `application/app_clock_sim.c`
   - `application/app_clock_core.c`
   - `application/app_clock.c`
   - `drivers/drv_key.c`
5. 按下面各用例的断点表设置断点。
6. 启动 `Debug`（Ctrl+F5 或菜单 Debug -> Start/Stop Debug Session）。
7. 程序会从 `User/main.c:14` 进入仿真分支 `AppClockSim_Main()` -> `AppClockSim_RunAll()`，并首先停在 `AppClockSim_RunTimeEditChecks()` 的第一个断点。
8. 使用 `Step Into (F11)` 和 `Step Over (F10)` 跟踪每个输入动作。

### 4.6 用例 A：进入改时 -> 切换字段 -> 加一 -> 减一 -> 取消

#### 输入 1：KEY1 click（进入改时）

仿真脚本触发位置：
- `app_clock_sim.c:106` — `AppClockSim_KeyClick(KEY_ID_KEY1);` 注入 KEY1 click

建议断点（按执行顺序，打在任何一处即可）：

**断点 A1-1** — 状态机收到 KEY1 click 事件（在 RUN 态下）：
```

行号: 554
源码:     if ((event.key_id == KEY_ID_KEY1) && (event.type == KEY_EVENT_TYPE_CLICK))
为什么: 这是 HandleKeyEvent 中 RUN 态处理 KEY1 click 的入口判断。在此处停下可以确认事件
        key_id=KEY_ID_KEY1(1)、type=KEY_EVENT_TYPE_CLICK(1)，且 clock->view=APP_VIEW_RUN(0)
观察点: event.key_id, event.type, clock->view
预期:   event.key_id == 1 (KEY_ID_KEY1), event.type == 1 (CLICK), clock->view == 0 (RUN)
```

**断点 A1-2** — 进入 EnterTimeEdit：
```
文件: application/app_clock_core.c
行号: 556
源码:         AppClockCore_EnterTimeEdit(clock, ui_dirty);
为什么: 确认 RUN 态 KEY1 click 确实路由到了 EnterTimeEdit
观察点: clock->view（执行前=0, 单步进入后应变为1）
预期:   执行 Step Into 进入 EnterTimeEdit
```

**断点 A1-3** — EnterTimeEdit 内部设置 view：
```

源码:     clock->view = APP_VIEW_SET_TIME_HOUR;
为什么: 确认 view 从 RUN(0) 切换为 SET_TIME_HOUR(1)

```
文件: application/app_clock_sim.c
行号: 107
源码:     AppClockSim_RefreshSnapshot();
为什么: 脚本在 KEY1 click 后立即抓取快照，下一行的 APP_CLOCK_SIM_CHECK 会验证结果

## 11. 已知局限与需要人类补充

1. **闹钟超时值**：当前实现为 30 秒（`APP_ALARM_RING_MS = 30000UL`），讲义要求为 60 秒。需确认是否为有意设计决定还是待修改项。 [需要人类补充]

2. **保存路径未纳入脚本**：`SET_TIME_SECOND -> save-time` 和 `SET_ALARM_HOUR -> SET_ALARM_MINUTE -> save-alarm` 两条保存路径的核心代码已存在，但仿真脚本尚未覆盖。如需补充，可参照现有脚本结构新增 `AppClockSim_RunSaveTimeCheck()` 和 `AppClockSim_RunSaveAlarmCheck()`。

3. **触摸逻辑键路径**：`AppClockCore_HandleTouchCommand()` 的代码已完整实现（`app_clock_core.c:587`），与物理键共用同一套状态机。但纯软件脚本中未直接注入触摸事件，如需覆盖可参照 `AppClockSim_KeyClick` 模式新增 `AppClockSim_TouchCommand()` 注入函数。

4. **红外路径**：`app_clock_remote.c` 中的 `AppClockRemote_HandleEvent()` 将红外事件映射为触摸逻辑命令，但纯软件脚本同样未覆盖。 [需要人类补充]

5. **硬件 RTC 与后备电池**：仿真模式使用软件 RTC（`Drv_Rtc_Task1ms()` 递增），不依赖 LSE 晶振和后备寄存器。真实硬件 RTC 的走时精度和断电保持仍需要真机验证。

6. **board_config.h 中的引脚定义**：按键实际 GPIO 端口和极性（`BOARD_KEY_ACTIVE_LOW`）由 `board_config.h` 中的宏决定。当前文档假设这些宏与硬件一致，但纯软件模式下这些宏不影响行为（因为走了 `APP_CLOCK_SIM_ENABLED` 分支）。
